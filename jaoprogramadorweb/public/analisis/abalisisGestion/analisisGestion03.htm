Análisis de Requisitos

Fases de análisis del sistema

El análisis de requisitos del sistema es igual que la fase de análisis en un sistema en métrica en esta fase lo que se realiza es definir los objetivos, es decir, que productos pretendemos obtener y definir que técnicas vamos a emplear para obtener estos productos.

Dentro de la metodología de desarrollo el análisis del sistema es la primera fase. En ella el analista más el cliente indicará las especificaciones es decir las necesidades de información y la arquitectura del sistema a desarrollar. Con el análisis de requisitos y las especificaciones funcionales tendremos información de los siguientes elementos:
1- Funcionalidad.
2- Restricciones.
3- Relaciones.
4- Costes.
5- Recursos.
6- Empleados.
7- Riesgos.
8- Tiempos para las diferentes tareas.
9- Personal que interviene en cada una de las fases.
10- Con todas estas informaciones tendremos una visión tanto del entorno físico como del tecnológico.


Objetivos
Los objetivos que se tienen que cumplir dentro de la fase de análisis nos van a venir dados por las siguientes necesidades:
1- las necesidades del cliente.
- Reuniones para saber las necesidades del cliente se realizaran reuniones y entrevistas entre los miembros del equipo de análisis el cliente y los usuarios finales.
- Restricciones económicas en temporales que va a dar el cliente.
- Productos existentes en el mercado.
Con estos tres elementos se realiza un documento conceptual del sistema este documento se ira modificando a medida que se obtienen los requisitos del sistema. Según la necesidad de realizar un estudio de viabilidad económica, técnica y legal.

Limitaciones económicas temporales: se tendrá que contestar a las siguientes preguntas:
1- ¿se terminará el proyecto a tiempo?
2- ¿Se terminará el proyecto con el dinero disponible?
3- ¿Con el personal disponible?
4- ¿Con las herramientas disponibles?
5- ¿Con las técnicas disponibles?

Viabilidad legal
Se estudiará
1- los elementos jurídicos relacionados con las actividades informáticas.
2- protección de datos.
3- obligaciones definidas en los contratos.
4- responsabilidades legales.

Con estos dos elementos las limitaciones económico temporales y la viabilidad legal se realizara un documento que la dirección del proyecto revisara para valorar la continuidad o el abandono del mismo.

Identificar y definir requisitos

Planificación de entrevistas, En cuanto a los objetivos


En cuanto a la especificación de requisitos. Los requisitos pueden ser de dos tipos:

- Funcionales: son aquellos que describen el comportamiento esperado del sistema.
- No funcionales: son aquellos que describen las propiedades e indican las restricciones, es decir, las limitaciones a las distintas posibilidades de trabajar los requisitos de funcionamiento indican las especificaciones de diseño también indican el tiempo de respuesta y ocupación de la memoria.
o Manejo de excepciones: todo lo que tiene que ver con el comportamiento no deseado del sistema.
Características que se esperan de una especificación:

- Completa: Hay que documentar todos los requisitos.
- Consistente: No puede haber requisitos contradictorios.
- Concisa: Tiene que escribirse con frases breves y fáciles de entender.
- No tiene que ser ambiguo, es decir no puede haber diferentes formas de entender una especificación.
- Verificable: Tiene que estar comprobada por el cliente para ver si es correcta.

Los principales defectos que aparecen en el manual de especificación:

- Trivialidades: Es decir, todo lo que sea obvio no hay que indicarlo.
- Ambigüedades: no se pueden utilizar frases que tengan más de un sentido.
- Omisiones: Hay que planear adecuadamente las entrevistas.
- Directrices de diseño e implantación.

Dentro de los requisitos funcionales se nos informará.

- Si se tiene que imprimir un albarán y las copias que se tienen que realizar.

Algunos de los requisitos no funcionales:
- la fecha de nacimiento no puede ser mayor que la fecha actual menos 16 años.
- Cualquier salario tiene que estar comprendido entre 150.000 y 300.000 pesetas.
- Especificación de hardware: se pretende aprovechar y Pentium 200.
- Software: se pretende utilizar W98.
- Referente al sistema: cualquier producto software que se utilice tiene que ser de Microsoft®.
- Cuando se produce un error se termine el programa.

1.3 Técnicas.

Las técnicas que se van a realizar son la realización de entrevistas y como consecuencia de ellas se realizaran los diagramas de flujos de datos.

ESPECIFICACION DE PROCESOS.

1. Diagramas de flujos de datos: estos diagramas solo representan el movimiento de los datos. Son una técnica grafica. Se utilizan sobre el análisis de requisitos. Sirven para representar las funciones, es decir, intervienen en la transformación de datos y transforman los datos a medida que se avanza en el sistema.


Entradas salidas

Movimiento de datos

a. Niveles de representación: estos diagramas representan un sistema a cualquier nivel de astricción. Los niveles de astricción o refinamiento permiten representar un mayor flujo de la información y mayor detalle funcional. Se suelen emplear hasta tres niveles. Es raro emplear niveles superiores a este y van numerados de 0 a N.

- Nivel 0: se conoce con el nombre de diagramas de contexto. Representan el flujo total de datos del sistema. La expansión de estos da origen a los diagramas de flujo de datos de Nivel 1 o diagramas de subsistemas.
- Nivel 1 y posteriores: un diagrama de datos de nivel 1 posee varios procesos conectados entre si mediante flechas. Cada proceso representa un subsistema y se crean niveles inferiores para obtener un mayor detalle.

b. Elementos de un diagrama de flujo:

- Entidades externas: se representan mediante un círculo o una elipse. Con el nombre en su interior, este nombre no puede ser un verbo. Son el origen y el destino de los datos. Producen información o la consumen. Son elementos externos al sistema. Pueden ser personas, Hardware o Software. No se pueden comunicar entre si, ya que el flujo de datos queda fuera del sistema, tampoco pueden relacionarse con un almacén de datos si no existe un proceso por medio. Se pueden repetir varias veces las entidades externas para que no existan entrecruzamientos. Pueden aparecer en cualquier nivel.

- Procesos: representan actividades que transforman la información. No hay correspondencia entre procesos y programas. Un proceso suele expandirse en múltiples programas, múltiples funciones o múltiples rutinas. Solo se representan los procesos contenidos en el sistema a modelar. Se les asigna como nombre o etiqueta un verbo que define la acción a realizar. Cada proceso tiene un flujo de datos de entrada y un flujo de datos de salida. No pueden ser ni origen ni destino. Se pueden conectar dos procesos entre si. La explosión o refinado de los procesos se realiza de uno en uno. Cuando explosionan procesos se detectan nuevos verbos y estos verbos significan nuevos procesos. Se llega hasta procesos primitivos, es decir, que no se pueden expandir más. En cada proceso primitivo se ponen sus características y esto se llama mini especificaciones. Las especificaciones se representan utilizando distintas técnicas.

Narrativa.
Ecuaciones matemáticas.
Tablas.
Diagramas.
Gráficos.
Descripciones en seudo códigos (lenguaje de descripción de programas).

Estas características sirven para crear los requisitos del software y para diseñar el programa que realizara el proceso.


- almacén de datos: sirven para el almacenamiento permanente o temporal de los datos. Pueden ser utilizados por procesos diferentes a los que los crean. Pueden ser utilizados por uno o más procesos. Tienen un flujo de datos de entrada y de salida. No puede haber comunicación entre almacenes de datos y entidades externas. Los almacenes de datos ni crean ni destruyen datos. Sobre ellos se realizan las operaciones de lectura, escritura y actualización. Solo aparecen en los diagramas de flujo que son utilizados, no aparecen en los diagramas de contexto.




- Flujos de datos: sirven para representar los movimientos de los datos. Se representan mediante flechas.

La punta de las flechas indica la dirección de los datos. Cada flecha posee un nombre o etiqueta que puede estar formada por un sustantivo y el participio del verbo del proceso que proviene.

Ejemplo: Un centro educativo desea crear un sistema de información que gestione las ausencias o retrasos a clase. Cada profesor pasa lista después de llegar a clase y escribe un parte de ausencias o retrasos por cada alumno que no este en el aula. Los datos del parte se introducen en el ordenador y después de las comprobaciones oportunas se envía al jefe de estudios un informe de las faltas diarias de los alumnos.
